# Skill Development Rules

> **SCOPE:** These rules apply ONLY when developing this skill itself (the skill-code-review repository). If this skill is installed inside another project (e.g., `someproject/.claude/skills/skill-code-review/`), this entire `.claude/` directory is out of scope — ignore it completely. Do not apply these rules to the containing project. Do not reference these files during code review dispatch.

## Architecture

**Wiki-tree routing for token efficiency:**

- The orchestrator (`code-reviewer.md`) reads `reviewers.wiki/index.md` and descends the tree semantically by `focus` strings; at leaf level the `activation:` block (file globs, structural signals, escalation) is the dispatch gate.
- The corpus has two layers: hand-authored sources at `reviewers.src/` (gitignored) and the wiki-organised tree at `reviewers.wiki/` produced by `skill-llm-wiki`. Authors only touch the source layer.
- Each leaf has rich frontmatter (`id`, `type`, `focus`, `covers[]`, `dimensions[]`, `audit_surface[]`, `activation`, `tools[]`, `tags[]`, `languages`) — that is the source of truth.
- The wiki layer handles clustering, slug generation, soft-DAG parents, balance enforcement, and root containment. Never edit `reviewers.wiki/` files by hand — every change is regenerated by a rebuild from `reviewers.src/`.
- Cross-cutting concerns (security, correctness, tests, docs, perf) get descended whenever the diff plausibly triggers them — there are no "always-on" universal reviewers any more.

## Adding a New Reviewer (or language / framework)

The procedure is the same for any leaf type — language reviewers are `lang-<name>.md`, frameworks are `fw-<name>.md`, plain reviewers use whatever stable kebab-case `id`.

1. Author `reviewers.src/<id>.md` with the v2 frontmatter (see `CONTRIBUTING.md` for the full schema).
2. Run the source validators:

   ```bash
   npm run validate:src
   npm run test:src
   ```

3. Rebuild the wiki via `skill-llm-wiki` (sibling project at `../skill-llm-wiki/` in this workspace):

   ```bash
   node ../skill-llm-wiki/scripts/cli.mjs build /absolute/path/to/skill-code-review/reviewers.src \
     --quality-mode deterministic --fanout-target 6 --max-depth 5 --soft-dag-parents --accept-dirty
   ```

4. Validate the rebuilt wiki, then rename `reviewers.src.wiki/` → `reviewers.wiki/`:

   ```bash
   node ../skill-llm-wiki/scripts/cli.mjs validate /absolute/path/to/skill-code-review/reviewers.src.wiki
   mv /absolute/path/to/skill-code-review/reviewers.wiki /tmp/reviewers.wiki.bak
   mv /absolute/path/to/skill-code-review/reviewers.src.wiki /absolute/path/to/skill-code-review/reviewers.wiki
   ```

5. Commit the source change AND the rebuilt wiki together. The wiki is the source of truth in the repo.

If adding a framework that the orchestrator doesn't yet recognise from manifests, also update Phase C of `code-reviewer.md` (the dependency-name → category table) so the Project Profile carries the right signal into Step 1's tree descent.

## Conventions

- **IDs**: kebab-case, must match filename. `lang-*` for languages, `fw-*` for frameworks, `sec-*` for security, `pattern-*` for design patterns, `antipattern-*` for anti-patterns, `arch-*` for architectural styles, `domain-*` for vertical concerns, `obs-*` for observability, `data-*` for data architecture, etc.
- **Tier limits**: see `scripts/lib/reviewer-schema.mjs` for per-type body length budgets — soft-warn on overrun, never hard-block.
- **Severity**: Critical (blocks merge), Important (blocks merge), Minor (advisory). The aggregator uses originating specialist's severity — it does not re-classify.
- **Dimensions**: every reviewer must declare ≥ 1 of: `architecture`, `correctness`, `documentation`, `performance`, `readability`, `security`, `tests`. Gate aggregation predicates against these.
- **Tags**: free-form topical tags. Used by gate predicates and by Tier-0 TF-IDF in clustering. Be specific and consistent — `state-management`, `circuit-breaker`, `solid` etc.
- **Tools**: optional `tools[]` array. Each entry: `name` (required), `command` (optional — exact command to run), `purpose` (required — what it checks). No install commands — AI resolves those at runtime.
- **Clickable file links**: in reports, use `[file:line](file#Lline)` format for GitHub/IDE compatibility.

## Before Every Commit

```bash
npm run validate:src && npm run test:src && npm run lint
```

The pre-commit hook enforces this automatically. After a wiki rebuild, also run `node ../skill-llm-wiki/scripts/cli.mjs validate <wiki>` and confirm 0 errors before committing the rebuilt tree.
