# Alembic Migrations

## When This Activates

Activates on diffs touching Alembic migration files under `versions/`, `alembic.ini`, or `env.py`. Alembic is the standard migration tool for SQLAlchemy projects, but its autogenerate feature misses important changes (indexes, constraints, enums), concurrent development creates branch conflicts, and offline mode produces subtly different SQL. This reviewer catches these Alembic-specific pitfalls.

## Audit Surface

- [ ] Autogenerated migration missing index creation or constraint change
- [ ] Multiple migration heads without alembic merge resolution
- [ ] Offline mode (--sql) used for migration that requires data inspection
- [ ] op.bulk_insert() or op.execute() for data migration in schema migration file
- [ ] Migration revision with incorrect down_revision breaking the chain
- [ ] env.py using hardcoded database URL instead of environment variable
- [ ] Missing op.batch_alter_table() for SQLite schema changes
- [ ] Autogenerate not detecting Enum type changes or column renames
- [ ] Migration performing table lock on large table without statement_timeout
- [ ] Missing create_constraint or drop_constraint for renamed constraints

## Detailed Checks

### Autogenerate Misses
<!-- activation: keywords=["autogenerate", "auto", "revision", "--autogenerate", "compare_type", "compare_server_default", "include_object", "render_item"] -->

- [ ] **Missing index detection**: flag autogenerated migrations that add columns used in queries but do not include corresponding `op.create_index()` -- autogenerate does not always detect index needs; review query patterns and add indexes manually
- [ ] **Enum changes missed**: flag Python Enum type changes in SQLAlchemy models without corresponding `op.execute("ALTER TYPE ... ADD VALUE ...")` in migration -- autogenerate does not detect Enum value additions; add them manually
- [ ] **Column rename as drop+add**: flag autogenerate output that drops a column and adds a new one instead of renaming -- autogenerate cannot detect renames; replace with `op.alter_column(... new_column_name=...)`
- [ ] **compare_type not enabled**: flag `env.py` without `compare_type=True` in `context.configure()` -- without this, autogenerate does not detect column type changes

### Branch Merging
<!-- activation: keywords=["heads", "merge", "branch", "down_revision", "head", "multiple", "conflict"] -->

- [ ] **Multiple heads**: flag Alembic migration tree with multiple heads (output of `alembic heads` shows >1) -- concurrent development creates branches; run `alembic merge` to create a merge revision before deploying
- [ ] **Incorrect down_revision**: flag migration with `down_revision` pointing to a non-existent or wrong parent -- this breaks the migration chain; verify with `alembic history`
- [ ] **Merge revision with operations**: flag merge revisions (created by `alembic merge`) that also contain schema operations -- merge revisions should be empty; put operations in a separate revision

### Offline Mode Limitations
<!-- activation: keywords=["offline", "--sql", "run_migrations_offline", "literal_binds", "script"] -->

- [ ] **Offline mode for data migration**: flag `--sql` (offline) generation for migrations that contain `op.execute()` with dynamic data or `op.bulk_insert()` -- offline mode renders SQL statically and cannot inspect data; run these migrations online
- [ ] **Missing literal_binds**: flag offline mode configuration without `literal_binds=True` in `context.configure()` -- without this, parameterized queries in offline SQL contain unresolved placeholders
- [ ] **Offline SQL not reviewed**: flag offline-generated SQL scripts applied to production without human review -- always inspect the generated SQL for database-specific syntax issues

### Data and Schema Separation
<!-- activation: keywords=["op.execute", "op.bulk_insert", "bulk_insert", "data", "UPDATE", "INSERT", "backfill", "RunPython"] -->

- [ ] **Data migration in schema file**: flag `op.execute("UPDATE ...")` or `op.bulk_insert()` in the same migration as `op.add_column()` or `op.alter_column()` -- separate data and schema operations into different revisions for independent rollback
- [ ] **Missing downgrade for data migration**: flag `op.execute("INSERT ...")` in upgrade without corresponding `op.execute("DELETE ...")` in downgrade -- data operations need explicit reversal
- [ ] **Large data backfill without batching**: flag `op.execute("UPDATE table SET ...")` without a WHERE clause limiting batch size -- large unbatched updates lock the table; use batched updates with row limits

### Environment Configuration
<!-- activation: keywords=["env.py", "config", "sqlalchemy.url", "target_metadata", "configure", "run_migrations"] -->

- [ ] **Hardcoded database URL**: flag `sqlalchemy.url` in `alembic.ini` or `env.py` containing a literal connection string with credentials -- use environment variables: `config.set_main_option("sqlalchemy.url", os.environ["DATABASE_URL"])`
- [ ] **Missing target_metadata**: flag `env.py` with `target_metadata = None` -- autogenerate needs model metadata to detect changes; set `target_metadata = Base.metadata`
- [ ] **Wrong schema binding**: flag `env.py` not setting `include_schemas=True` for multi-schema databases -- Alembic only inspects the default schema unless configured otherwise

### SQLite Batch Mode
<!-- activation: keywords=["batch_alter_table", "SQLite", "sqlite", "batch", "alter_column", "drop_column", "rename"] -->

- [ ] **Missing batch mode for SQLite**: flag `op.alter_column()`, `op.drop_column()`, or constraint changes without `op.batch_alter_table()` on SQLite databases -- SQLite does not support most ALTER TABLE operations; batch mode recreates the table
- [ ] **Batch mode on non-SQLite**: flag `op.batch_alter_table()` on PostgreSQL or MySQL -- batch mode is unnecessary overhead for databases with full ALTER TABLE support

## Common False Positives

- **Intentional drop+add**: when a column truly needs a type change that requires dropping and recreating, autogenerate's output is correct.
- **Offline SQL for CI**: generating offline SQL for review in CI without executing it is a valid workflow.
- **Data migration with schema**: small data fixes (e.g., backfilling a default value for a new NOT NULL column) in the same migration as the schema change is acceptable when tightly coupled.

## Severity Guidance

| Finding | Severity |
|---|---|
| Hardcoded database URL with credentials | Critical |
| Multiple unresolved heads in production migration chain | Critical |
| Broken down_revision chain | Important |
| Autogenerate missing column rename (drop+add instead) | Important |
| Data migration mixed with schema migration | Important |
| Offline mode for data-dependent migration | Important |
| Missing compare_type in autogenerate config | Minor |
| Large data backfill without batching | Important |
| Missing batch mode for SQLite | Minor |
| Merge revision containing operations | Minor |
